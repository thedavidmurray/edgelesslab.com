1e:T1546,<p>Most AI agents die when you close the laptop. Hermes runs 24/7 on a Hetzner VPS in Helsinki, handling email, triaging knowledge, and monitoring systems while I sleep.</p>
<h2 id="the-problem-with-session-based-agents">The Problem With Session-Based Agents</h2>
<p>Every AI coding assistant has the same limitation: it exists inside your session. Close the terminal, lose the agent. Come back tomorrow, re-explain everything.</p>
<p>I needed an agent that processes incoming signals on a schedule, triages my knowledge base without me watching, monitors system health and only bothers me when something breaks, and remembers what it learned yesterday.</p>
<p>So I built Hermes. It runs on a $5/month Hetzner VPS in Helsinki, and it hasn't needed a manual restart in three months.</p>
<h2 id="what-hermes-actually-does">What Hermes Actually Does</h2>
<p>Hermes isn't a chatbot. It's a Chief of Staff that runs 8 cron jobs autonomously:</p>
<p><strong>Every 6 hours:</strong> Health check across all services. If something is down, I get a Telegram message. If everything is fine, silence. This is the most important design decision: agents should only interrupt you when something needs attention.</p>
<p><strong>Every 4 hours:</strong> Email triage. Reads incoming email, categorizes by urgency, drafts responses for routine messages, flags anything that needs human judgment.</p>
<p><strong>Daily at 4pm UTC:</strong> Newsletter digest. Processes accumulated RSS and email newsletters, extracts signal, scores by relevance (1-10), writes summaries to the knowledge base.</p>
<p><strong>Twice weekly:</strong> Dream consolidation. Reviews everything it learned that week, identifies patterns, promotes high-value insights from inbox to permanent knowledge storage.</p>
<h2 id="the-architecture">The Architecture</h2>
<ul><li><strong>Model:</strong> Kimi K2.5 via Fireworks AI (flat-rate, unlimited tokens)</li><li><strong>Communication:</strong> Telegram bot for human interaction, file-based inbox for agent-to-agent dispatch</li><li><strong>Memory:</strong> Flat-file MEMORY.md + Obsidian vault via rsync</li><li><strong>Skills:</strong> 91 custom skills covering web research, email, GitHub, code review, knowledge curation</li><li><strong>Tools:</strong> 102 available tools including DuckDuckGo, Perplexity API, GitHub CLI, file system</li></ul>
<p>The VPS costs $5.35/month. Model inference costs about $4-5/week via Fireworks. Total: roughly $26/month for an always-on AI operations team member.</p>
<h2 id="the-soul-md-file">The SOUL.md File</h2>
<p>Every Hermes session loads a personality file called SOUL.md. It contains the behavioral rules that make Hermes useful instead of annoying:</p>
<ol><li>Lead with the answer, not the reasoning</li><li>For cron jobs: only message David if something needs attention</li><li>Do NOT log routine status checks to memory</li><li>Web search priority: DuckDuckGo first, then Perplexity, then memory recall</li><li>Decision tree: local data first, then dispatch to Mac, then ask David</li></ol>
<p>Rule 2 is the most important. A naive agent sends you a message every time it completes a cron job. Twelve times a day. That's not helpful, that's spam. Hermes only talks to me when something is wrong.</p>
<h2 id="three-communication-channels">Three Communication Channels</h2>
<p><strong>Telegram</strong> is for quick questions. Conversational, one-shot, no tool use. Fast and lightweight.</p>
<p><strong>Direct API</strong> is for programmatic access. Scripts hit the chat completions endpoint for structured responses.</p>
<p><strong>Inbox dispatch</strong> is for real work. Drop a markdown directive in a shared folder, rsync carries it to the VPS, Hermes processes it as an autonomous task with the full toolset. Round trip is about 17 minutes worst case.</p>
<p>The inbox pattern is critical. Chat interfaces encourage chat behavior. When you want an agent to actually do work, give it a work order, not a conversation.</p>
<h2 id="what-breaks-and-how-it-recovers">What Breaks (And How It Recovers)</h2>
<p><strong>Confabulation.</strong> Hermes once claimed it had created an entire wiki structure on the server. None of it existed. The API is stateless. Fix: always verify file claims independently.</p>
<p><strong>Provider routing confusion.</strong> The auth state silently overrode the config file. Hermes used the wrong model for weeks. Fix: provider selection now goes through a single code path with explicit logging.</p>
<p><strong>Inbox self-messaging.</strong> An early dispatch system could create loops where Hermes dispatched tasks to itself. Fix: a self-message guard drops any directive where <code>from == to</code>.</p>
<p>Each failure led to a targeted fix. Not a framework rewrite. Just a guard clause in the right place.</p>
<h2 id="the-compound-value">The Compound Value</h2>
<p>After three months of continuous operation, Hermes has processed 3,200+ documents into the knowledge base, triaged 8,000+ tasks without a manual restart, caught 14 system issues before they became problems, and built a knowledge graph I search daily.</p>
<p>The value isn't in any single cron job. It's in the compound effect of an agent that runs while you don't. Knowledge accumulates. Patterns emerge. The system gets smarter not because the model improves, but because the data it operates on gets richer.</p>
