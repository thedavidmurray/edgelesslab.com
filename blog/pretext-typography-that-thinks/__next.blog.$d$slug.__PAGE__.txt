4:T15db,<p>CSS gives you two options for text layout: a block that fills its container, or <code>fit-content</code> that shrinks to the longest line. Neither lets you answer "how tall will this paragraph be at 320px wide?" without rendering it first.</p>
<p>PreText answers that question in 0.002ms, before a single DOM node exists. That changes what's possible.</p>
<h2 id="the-measurement-gap">The Measurement Gap</h2>
<p>Every masonry layout, every accordion animation, every balanced text block on the web has the same problem: you need to know the height of something before you render it.</p>
<p>The standard approach is render-measure-rerender. Mount the DOM, read <code>offsetHeight</code>, reposition. This causes layout thrash: visible flicker where elements jump as the browser recalculates.</p>
<p>PreText skips the DOM entirely. It uses the Canvas 2D text measurement API to calculate exact line breaks, line widths, and total height for any text at any width. The results match browser rendering because they use the same font metrics.</p>
<h2 id="what-we-built-with-it">What We Built With It</h2>
<p>The <a href="/">Edgeless Lab site</a> uses PreText in six places, each solving a different layout problem:</p>
<p><strong>Masonry product grid</strong>: The <a href="/products">products page</a> lays out product cards in a masonry grid. Each card's height is different because descriptions vary in length. PreText measures every description, calculates the exact card height, and places cards using a shortest-column algorithm. Zero DOM measurement. Zero layout shift.</p>
<p><strong>Shrink-wrap balanced text</strong>: The about section on the homepage wraps text to the tightest possible width that preserves line count. CSS <code>fit-content</code> leaves dead space on the last line. PreText's <code>walkLineRanges</code> finds the actual maximum line width, giving text a balanced, typeset appearance.</p>
<p><strong>Hero cursor reflow</strong>: The homepage subtitle text flows around your cursor in real time. As you move the mouse, PreText recalculates line breaks around a circular obstacle at 60fps using <code>layoutNextLine</code> with remaining-width budgets. Pure DOM manipulation, no React re-renders.</p>
<p><strong>Stagger reveal</strong>: The stack section reveals text line-by-line on scroll. PreText's <code>layoutWithLines</code> returns exact line widths, so wider lines slide further during the entrance animation, creating geometry-driven stagger.</p>
<p><strong>Rich inline flow</strong>: The stack pipeline displays tool names in monospace and descriptions in sans-serif, reflowing as a single mixed-font paragraph. Each segment is measured separately; <code>layoutNextLine</code> coordinates the width budget across font changes.</p>
<p><strong>Generative ASCII art</strong>: The <a href="/lab/generative-ascii">generative ASCII experiment</a> uses PreText to measure character widths for proportional-to-monospace mapping, ensuring spatial accuracy in typographic art.</p>
<h2 id="the-api-in-30-seconds">The API in 30 Seconds</h2>
<p>PreText exposes six functions. You only need two for most work:</p>
<p><code>prepare(text, font)</code>: tokenizes text and measures segment widths. Returns a prepared object. Runs once per text/font pair.</p>
<p><code>layout(prepared, width, lineHeight)</code>: calculates total height and line count at a given container width. Returns <code>{ height, lineCount }</code>. Runs in microseconds.</p>
<p>For advanced layouts:</p>
<p><code>layoutWithLines(prepared, width, lineHeight)</code>: returns every line with its exact pixel width. Use for stagger animations or justified text.</p>
<p><code>walkLineRanges(prepared, font, lineHeight, callback)</code>: iterates line ranges for binary search over widths (shrink-wrap).</p>
<p><code>layoutNextLine(prepared, cursor, maxWidth, lineHeight)</code>: advances one line at a time. Use for multi-column, obstacle avoidance, or mixed-font layouts.</p>
<p><code>prepareWithSegments(text, font)</code>: like prepare, but returns individual segment widths for character-level operations.</p>
<h2 id="why-this-matters-for-product-pages">Why This Matters for Product Pages</h2>
<p>When you're selling developer tools, the site itself is a portfolio piece. A masonry layout that loads without flicker. Accordion animations that hit their target height on the first frame. Text that reflows around your cursor without a single layout recalculation.</p>
<p>These aren't features for their own sake. They demonstrate the kind of engineering precision that the products represent. If the site can't get typography right, why would you trust the templates?</p>
<h2 id="getting-started">Getting Started</h2>
<p>PreText is an npm package: <code>@chenglou/pretext</code>. It's 4KB gzipped, zero dependencies, works in any framework. The <a href="https://chenglou.me/pretext/">PreText demos</a> show every technique in isolation.</p>
<p>The integration pattern: load PreText in a <code>useEffect</code>, wait for fonts, then measure. Server-side, fall back to CSS estimates. The switch from fallback to measured layout is imperceptible because the content is identical; only the positioning changes.</p>
<pre><code>const { ready, prepare, layout } = usePreText("Geist");

if (ready) {
  const prepared = prepare(text, '14px "Geist"');
  const { height } = layout(prepared, containerWidth, 22.4);
  // height is exact, before any DOM exists
}</code></pre>
