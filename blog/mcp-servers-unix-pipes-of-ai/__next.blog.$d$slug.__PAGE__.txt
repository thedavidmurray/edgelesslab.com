4:T1758,<p>In 1978, Doug McIlroy wrote the Unix philosophy in three sentences. The one that matters: "Write programs that do one thing and do it well. Write programs to work together."</p>
<p>Forty-eight years later, we're rediscovering this idea for AI agents, and calling it MCP.</p>
<h2 id="what-mcp-actually-is">What MCP Actually Is</h2>
<p>The Model Context Protocol is a JSON-RPC spec that lets AI models call external tools through a standardized interface. An MCP server exposes a list of tools. A client (like Claude Code) connects to those servers and gets access to those tools. The model calls them by name with arguments, and gets back structured results.</p>
<p>That's it. No custom integrations per model. No bespoke SDKs. Define your tool once, and any MCP-compatible client can use it.</p>
<p>Sound familiar? It should. It's stdin/stdout with better types.</p>
<h2 id="the-unix-parallel">The Unix Parallel</h2>
<p>The power of Unix pipes wasn't any individual tool; it was composability. <code>cat file | grep pattern | sort | uniq -c</code> does something none of those tools could do alone. The protocol (text on stdout/stdin) made composition possible without any of the tools knowing about each other.</p>
<p>MCP does the same thing for AI tools. The protocol is JSON-RPC over stdio (or HTTP). The tools are small, focused, independently deployable. The composition happens in the model's reasoning layer instead of a shell.</p>
<p>The key insight in both cases: <strong>the protocol is the product</strong>. Not any single tool. Not any particular capability. The protocol is what enables the ecosystem.</p>
<h2 id="what-this-looks-like-in-practice">What This Looks Like in Practice</h2>
<p>The <a href="/blog/building-ai-agent-infrastructure-solo">agent infrastructure at Edgeless Lab</a> runs several MCP servers. Each one does one thing:</p>
<p><strong>ChromaDB search server</strong>: semantic search across a knowledge base of 7,000+ documents. Takes a query string, returns ranked results with similarity scores. That's the whole API.</p>
<p><strong>Obsidian vault query server</strong>: read and search the Obsidian vault by tag, folder, or full-text. Agents can retrieve specific notes or scan for relevant context without touching the filesystem directly.</p>
<p><strong>Backlog management server</strong>: read and write tasks in a structured backlog. Lets agents file their own tasks, check status, and mark things complete. The backlog is a text file format; the MCP server is the typed interface over it.</p>
<p><strong>Inter-agent messaging server</strong>: a pub/sub channel for agents to send messages to each other. An orchestrator agent can dispatch work; worker agents can report back. Real-time, without a message queue.</p>
<p>None of these tools know about each other. Any agent can use any combination. Add a new server and it's immediately available to every agent in the system.</p>
<h2 id="why-libraries-were-the-wrong-model">Why Libraries Were the Wrong Model</h2>
<p>Before MCP, tool access meant libraries. You'd import the Anthropic SDK, write a tool schema, register it with the client. Then repeat for every model you wanted to support. When OpenAI updated their function calling format, you'd update every integration.</p>
<p>This created tight coupling between your tools and your model provider. Switching models meant rewriting integrations. Testing a tool meant testing it inside a model's context.</p>
<p>MCP decouples these completely. The server doesn't know what model is calling it. The model doesn't care how the server is implemented. The server could be TypeScript, Python, Go. Doesn't matter. The protocol is the boundary.</p>
<p>This is exactly what made Unix pipes powerful: <code>grep</code> doesn't know it's receiving input from <code>cat</code>. It just reads from stdin.</p>
<h2 id="tools-as-services-not-libraries">Tools as Services, Not Libraries</h2>
<p>The shift MCP enables is treating tools as services rather than library calls. This changes the operational model significantly:</p>
<p>You can <strong>deploy tools independently</strong>. The Obsidian server runs on your Mac (it needs local filesystem access). The ChromaDB server runs wherever ChromaDB is. The trading data server runs on the VPS. Each deployed where it makes sense.</p>
<p>You can <strong>test tools independently</strong>. An MCP server is just an HTTP or stdio server. You can test it with <code>curl</code> or any JSON-RPC client. No model required.</p>
<p>You can <strong>version tools independently</strong>. Update the ChromaDB server without touching anything else. Agents pick up the new tool definition on next connection.</p>
<p>You can <strong>compose without coordination</strong>. Agent A uses the vault server and the backlog server. Agent B uses the vault server and the messaging server. Neither knows about the other, but they share infrastructure.</p>
<h2 id="the-ecosystem-implication">The Ecosystem Implication</h2>
<p>The deeper implication is that MCP creates a tool ecosystem that outlives any particular model or provider. A tool you write today for Claude will work with whatever succeeds Claude. The investment is in the tool, not the integration.</p>
<p>This is how Unix tools from the 1970s still run on your Mac today. The protocol survived everything else changing.</p>
<p>MCP is early. The tooling is rough in places. Server discovery is manual. Error handling is inconsistent. But the protocol is right, and the ecosystem will follow.</p>
<p>For anyone building agent infrastructure: start treating your tools as MCP servers, not library functions. The composition benefits compound quickly, and you're building on the right abstraction for the next decade of AI development.</p>
