4:T13f1,<p>AI agents confabulate. They claim files exist that don't. They report tasks complete that aren't. The grounding problem isn't philosophical. It's operational, and it will cost you hours if you don't solve it.</p>
<h2 id="the-confabulation-incident">The Confabulation Incident</h2>
<p>Hermes, my 24/7 agent running on a VPS in Helsinki, told me it had created a comprehensive wiki structure at <code>04-Wiki/</code> on the server. Four directories, twelve files, cross-referenced with the knowledge base.</p>
<p>None of it existed.</p>
<p>The Hermes API is stateless. Between sessions, it has no memory of what it has or hasn't done. When asked about prior work, it does what any language model does: it generates a plausible answer. The answer sounded exactly like something Hermes would have done. It just hadn't.</p>
<p>This is the grounding problem for production agents. Not "can an AI understand the real world?" but "can your agent distinguish between what it did and what it could have done?"</p>
<h2 id="why-agents-lie-unintentionally">Why Agents Lie (Unintentionally)</h2>
<p>There are three failure modes:</p>
<p><strong>Confabulation.</strong> The agent generates plausible descriptions of work it never performed. This happens most often when you ask about past actions in a stateless system.</p>
<p><strong>Premature completion.</strong> The agent reports a task as done based on partial evidence. "I updated the file" when the write failed silently. "The test passes" when it ran a different test. The agent isn't lying. It's pattern-matching on what "done" usually looks like.</p>
<p><strong>Scope drift.</strong> The agent does real work, but not the work you asked for. It optimizes an adjacent function instead of fixing the bug. The work is real, verified, and wrong.</p>
<h2 id="the-grounding-stack">The Grounding Stack</h2>
<p>After six months of running autonomous agents, here's the grounding stack that actually works:</p>
<p><strong>Layer 1: Verify, don't trust.</strong> Every claim an agent makes about the file system gets verified by a separate process. "I created the file" gets <code>ls -la</code>. "The service is running" gets <code>curl localhost:port/health</code>. This sounds tedious. It's the single most important practice in agent operations.</p>
<p><strong>Layer 2: Evidence-based completion.</strong> Agents cannot declare a task complete without providing evidence. A passing test. A file that exists. A command that returns 0.</p>
<pre><code>EVIDENCE_CHECKS = {
    "test": lambda path: subprocess.run(["pytest", path]).returncode == 0,
    "file_exists": lambda path: os.path.exists(path),
    "command": lambda cmd: subprocess.run(cmd, shell=True).returncode == 0,
}</code></pre>
<p><strong>Layer 3: Grounding packets.</strong> At the start of every session, a grounding document loads the current state of the workspace. Not what the agent remembers, but what actually exists right now. File trees, service status, recent git log, active tasks.</p>
<p><strong>Layer 4: Self-message guards.</strong> Agents that can dispatch tasks need loop protection. A depth counter caps recursive dispatch. A <code>from == to</code> guard prevents self-messaging.</p>
<h2 id="the-memory-contract">The Memory Contract</h2>
<p>Hermes reads from four layers of memory, each with different trust levels:</p>
<ul><li><strong>Episodic ledger</strong> (SQLite, append-only): High trust. What actually happened.</li><li><strong>Semantic index</strong> (ChromaDB vectors): Medium trust. Search results may be stale.</li><li><strong>Curated vault</strong> (Obsidian markdown): High trust. Human-reviewed.</li><li><strong>Agent memory</strong> (MEMORY.md flat file): Low trust. May contain confabulated entries.</li></ul>
<p>The key insight: not all memory is equally trustworthy. Treating all memory sources as equally reliable is how you get agents acting on bad information.</p>
<h2 id="recovery-patterns">Recovery Patterns</h2>
<p>When an agent gets grounded incorrectly, recovery has three steps:</p>
<ol><li><strong>Detect the divergence.</strong> Usually via a failing verification check or a human noticing something off.</li><li><strong>Reload from ground truth.</strong> Don't try to "correct" the agent's beliefs. Kill the session, regenerate the grounding packet from actual system state, start fresh.</li><li><strong>Add a guard for this specific failure.</strong> Each grounding failure reveals a gap. The <code>04-Wiki/</code> incident led to a post-dispatch file existence check. Each guard is simple, specific, and permanent.</li></ol>
<h2 id="the-operational-discipline">The Operational Discipline</h2>
<p>Running autonomous agents is 20% building and 80% operational discipline. The agents don't get smarter on their own. They get more reliable because you add guards, verify outputs, and grind down the failure modes one at a time.</p>
<p>The model will hallucinate. The file system will have race conditions. The network will fail. The question isn't whether your agent will get grounded incorrectly.
