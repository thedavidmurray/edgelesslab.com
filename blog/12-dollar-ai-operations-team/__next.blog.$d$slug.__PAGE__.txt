<h2 id="the-agent-topology">The Agent Topology</h2>
<p>Five agents run in a dispatch/worker topology. This isn't decorative. It's the simplest structure that solves the actual production problems.</p>
<div class="blog-viz blog-flow"><div class="blog-viz-title">Agent Topology</div><div class="blog-flow-row"><span class="blog-flow-box">Human</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Hermes (CoS)</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Dispatch (COO)</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Builder</span></div><div class="blog-flow-row"><span class="blog-flow-box">Dispatch (COO)</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Researcher</span></div><div class="blog-flow-row"><span class="blog-flow-box">Dispatch (COO)</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Verifier</span></div></div>
<p><strong>Hermes (Chief of Staff)</strong> — My primary session agent. Receives all human requests, decides whether to execute directly or delegate. Runs on Kimi K2.5 via Fireworks. Context window management is the constraint: it sees the full project state and delegates to specialists when the task requires specific tools or extended processing.</p>
<p><strong>Dispatch (COO)</strong> — A Paperclip-managed agent that never executes tasks directly. Its only job is routing: receive task requests from Hermes, assign to appropriate workers, track state machine transitions, escalate stuck tasks. It runs on a lighter model (DeepSeek V3.2) because its cognitive load is lower—it's matching patterns, not reasoning about code.</p>
<p><strong>Builder</strong> — Claude Code agent on a VPS. Handles all code changes: feature implementation, bug fixes, refactoring. Runs on Anthropic Claude via standard API. The VPS isolates it from my local machine state, which means it can run overnight without my laptop being open.</p>
<p><strong>Researcher</strong> — Deep research agent. Consumes RSS feeds, YouTube transcripts, arXiv papers, synthesizes findings into structured reports. Runs Kimi K2.5 with extended context. Its output feeds directly into the knowledge base.</p>
<p><strong>Verifier</strong> — Quality control agent. Reviews Builder's output, runs tests, checks for security issues, validates against acceptance criteria. Acts as a gate before deployment.</p>
<p>The topology solves three specific failure modes I've hit with single-agent approaches:</p>
<ol><li><strong>Context pollution</strong>: When a single agent switches between coding and research, it drops relevant context from the earlier task. Specialists keep their context focused.</li><li><strong>Tool confusion</strong>: Agents with 20+ tools start calling the wrong ones. Specialists have 4-6 tools each. The tool selection accuracy is visibly higher.</li><li><strong>State loss on crash</strong>: A single long-running session that crashes loses everything. Distributed state means any single agent can restart without losing the system's progress.</li></ol>

<p>Kimi K2.5 handles 70% of tasks because it's the cheapest generalist that doesn't hallucinate tools. DeepSeek takes the high-volume, low-cognitive-load work (formatting, simple transformations, status checks). Claude Opus is reserved for security-sensitive reviews—it's expensive but catches issues the cheaper models miss. Codex handles bulk code generation where context length matters more than nuance.</p>
<p>The routing decision happens at the dispatch layer. Tasks include a complexity tag (low/medium/high) and a security flag. Low complexity + no security flag → DeepSeek. High complexity or security flag → Kimi or Opus depending on domain.</p>
<p>This routing alone cuts costs 10x versus using a single model for everything.</p>
<h2 id="the-knowledge-base-loop">The Knowledge Base Loop</h2>
<p>Every agent operation feeds a knowledge base. Not as an afterthought—as a core system function.</p>
<div class="blog-viz blog-flow"><div class="blog-viz-title">Knowledge Base Loop</div><div class="blog-flow-row"><span class="blog-flow-box">Capture</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Synthesize</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Verify</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Inject</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Agents</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Capture</span></div></div>
<p>The loop works like this:</p>
<ol><li><strong>Capture</strong>: All agent outputs, research findings, error logs, and human corrections write to ChromaDB with embeddings.</li><li><strong>Synthesize</strong>: A nightly batch job (separate agent) queries for related documents, detects themes, and writes synthesized summaries.</li><li><strong>Verify</strong>: Another agent samples the synthesized notes, checks for factual drift or contradictions, flags issues.</li><li><strong>Inject</strong>: The verified synthesis becomes retrievable context for all agents.</li></ol>
<p>The loop means agents don't just have tools—they have memory of what the system has learned. When Builder encounters an error, it can query: "how did we solve similar errors before?" The answer comes from actual previous sessions, not generic training data.</p>
<p>The KB infrastructure costs $2.80/week (self-hosted ChromaDB on a €6.50 Hetzner instance). The managed equivalent (Pinecone, Weaviate Cloud) runs $200-400/month.</p>
<h2 id="file-based-agent-communication">File-Based Agent Communication</h2>
<p>The agents communicate through two channels:</p>
<p><strong>Agent Bus</strong>: Real-time MCP connection on port 9800. Handles urgent coordination: task assignment, status updates, human escalation. Messages route through a local daemon that queues for offline agents.</p>
<p><strong>Async Inboxes</strong>: File-based system synced via rsync every 60 seconds. Each agent has an inbox directory. Dispatch writes task files, workers read and write results. The filesystem is the message queue.</p>
<p>Why files instead of a proper message queue (RabbitMQ, Redis)?</p>
<ol><li><strong>Observability</strong>: I can <code>cat /inbox/builder/task-4821.json</code> and see exactly what was sent.</li><li><strong>Recovery</strong>: If a task fails, the file is right there with full context. No log archaeology.</li><li><strong>Zero ops</strong>: No database to manage, no connection pools, no retry logic. The filesystem has been reliable for 50 years.</li></ol>
<p>The latency is higher (60s sync cycle) but the reliability is perfect. For tasks that need real-time, the Agent Bus handles it. Most tasks are fine with 60s latency.</p>
<h2 id="the-failure-modes">The Failure Modes</h2>
<p>Cheap infrastructure has specific failure modes. I've hit them.</p>
<p><strong>Session poisoning</strong>: An agent corrupted its own skill definitions through repeated partial updates. The corruption spread to other agents that read the shared skill file. Detection took 6 hours. Recovery required restoring from backup and adding versioned skill files.</p>
<p>The fix: skill files now include a checksum. Agents verify before loading. Corrupted skills fail closed (agent stops) rather than open (agent runs with bad definitions).</p>
<p><strong>Model degradation</strong>: Kimi K2.5 had a quality regression on one Fireworks deployment. The routing layer detected elevated error rates and automatically shifted load to the backup deployment. Total impact: 4 minutes of degraded service.</p>
<p>The fix: health checks on model endpoints, automatic failover, circuit breaker pattern for failing providers.</p>
<p><strong>Resource exhaustion</strong>: ChromaDB hit its memory limit during a large embedding batch. The indexer kept retrying, filling logs, failing silently. The KB synthesizer agent detected the backlog growth and alerted before the disk filled.</p>
<p>The fix: resource-aware batch sizing, explicit memory limits, monitoring on queue depth not just error rates.</p>
<h2 id="what-this-architecture-can-t-do">What This Architecture Can't Do</h2>
<p>Honest limitations:</p>
<ul><li><strong>No high-availability guarantee</strong>: Single VPS, single Mac. If Hetzner has an outage, the remote agents stop. Recovery is manual.</li><li><strong>No multi-region redundancy</strong>: 60s rsync is fine for async tasks, but real-time coordination can't survive a network partition.</li><li><strong>No formal verification</strong>: The state machine is tested, not proven. Edge cases exist.</li><li><strong>No enterprise compliance</strong>: No SOC 2, no audit logs for regulators. This is a solo operation.</li></ul>
<p>The architecture optimizes for "good enough for one person" not "good enough for 1,000 customers."</p>
<h2 id="getting-started">Getting Started</h2>
